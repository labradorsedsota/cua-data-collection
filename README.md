# 项目03 — 从 GitHub 找 CUA 数据

## 一、项目定位

**一句话目标：** 用 mano-cua 对 GitHub 开源项目的已知 UI Bug 进行大规模测试，采集高质量 CUA 操作轨迹数据。

**核心逻辑：** 已修复的 Bug = ground truth（已知答案的考试）。checkout 到修复前的 buggy 版本 → 部署 → mano-cua 盲测 → 记录是否命中 + 操作轨迹。

**数据需求方：** 老傅（FTY），CUA 模型负责人。

**与之前项目的区别：**
之前（项目01/02）采集 CUA 轨迹的方式是「自己做应用 → 自己写 test case → mano-cua 执行」，全链路自建。本项目改用开源方案——应用和 Bug 直接取自 GitHub 开源项目，不再自己造应用和写 case。两条路径的终点相同：获取 mano-cua 的操作轨迹数据。新路径的优势是规模化快、数据多样性好、不依赖自研应用。

---

## 二、全链路流程

```
Phase 1: 筛选准备
  ① 筛选项目 → ② 筛选 Bug → ③ 构造任务清单

Phase 2: 批量执行
  ④ 部署 buggy 版本 → ⑤ 执行 mano-cua 测试 → ⑥ 收集结果

Phase 3: 汇总交付
  ⑦ 汇总报告 → ⑧ 交付老傅
```

## 四、10 Agent 编排方案

### 4.1 架构设计

推荐采用 **「1 调度 + N 执行」** 模式：

```
                    ┌─────────────┐
                    │   Pichai    │ ← 项目管理 & 调度
                    │  (PM/调度)   │
                    └──────┬──────┘
                           │ 任务分发 / 进度汇总
              ┌────────────┼────────────┐
              │            │            │
        ┌─────┴─────┐ ┌───┴───┐ ┌─────┴─────┐
        │ Worker 1  │ │  ...  │ │ Worker 10 │
        │ (执行Agent)│ │       │ │ (执行Agent)│
        └───────────┘ └───────┘ └───────────┘
              │                       │
              └───────┬───────────────┘
                      ↓
              智子（质量抽检 & 汇总）
```

### 4.2 每个 Worker 的执行单元

一个 Worker 接收一个 **任务包**，任务包内容：

```yaml
task_id: "CUA-001"
project: "ivegamsft/luxesite"
bug_issue: "#258"
buggy_commit: "39a6e11"        # fix PR 的 parent commit
deploy_command: "pnpm install && pnpm dev"
test_description: "打开推荐页，点击翻页按钮逐页浏览，检查切换过程是否正常流畅"
expected_url: "http://localhost:3000"
```

Worker 执行流程：
1. Clone 项目 → checkout buggy_commit
2. 执行 deploy_command → 确认服务启动
3. 调用 mano-cua 执行测试（传入 test_description + URL）
4. 收集结果：sess-id、是否复现、步数
5. 上报结果到 PM

### 4.3 任务分配策略

**推荐按项目分组：** 同一个项目的多个 Bug 分配给同一个 Worker。

原因：
- 避免重复 clone/部署（同项目不同 Bug 可能在同一个 buggy commit）
- Worker 对项目越熟悉，执行越稳定
- 如果项目部署有坑，只影响一个 Worker，其他 Worker 不被阻塞

**任务量预估：**
- 智子筛选出的候选项目约 10-20 个
- 每个项目 15-50 个可测 Bug
- 总任务量预估 200-500 个 case
- 10 个 Worker，每个 Worker 分配 20-50 个 case

### 4.4 通信机制

**方案 A：DMWork 群消息（推荐起步阶段用）**
- 建一个「CUA 执行群」，所有 Worker + PM 在群内
- PM 发任务包 → Worker 回报结果
- 优点：可追溯、人类可观测
- 缺点：消息量大时嘈杂

**方案 B：文件 + API 驱动（推荐规模化阶段用）**
- 任务清单放在共享 repo（如 GitHub）
- Worker 通过 API 领取任务、提交结果
- PM 通过 repo 汇总
- 优点：结构化、可并行、不依赖消息
- 缺点：需要额外搭建

**建议：先 A 后 B**——先用 DMWork 群跑通 10 个 case 验证流程，再切到结构化方案跑量。

## 五、质量控制

### 5.1 数据质量标准（需老傅确认）

| 维度 | 标准 |
|------|------|
| 轨迹完整性 | mano-cua session 正常结束（非中断/超时） |
| 任务描述质量 | 中文、不暴露 bug、只描述要验证的功能点 |
| 复现判定 | 明确标注：✅ 复现 / ⚠️ 部分复现 / ❌ 未复现 |
| 产出格式 | 项目名称 \| 测试内容 \| sess-id \| 是否复现 |

### 5.2 PM 质量把控手段

1. **首轮抽检**：每个 Worker 的前 3 个 case，PM 检查 sess-id 对应的轨迹
2. **异常监控**：超时率 > 30% 或未复现率 > 70% 的 Worker 暂停排查
3. **每日汇总**：每天向老傅 & Emily 同步进度
